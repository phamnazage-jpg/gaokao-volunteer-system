## 2. 产品概述

### 2.1 产品定位

**一句话描述**：为高考考生提供专业化、数据驱动、政策合规的志愿填报辅助的AI系统。

**当前项目定位**：人工服务运营增强系统；用户端 Web 自助闭环仅为本地 MVP/目标态，不应按完整 Web 自助 SaaS 对外承诺。

> 2026-06-19 校准：删除/匿名化保留期门禁（E02002）已落地为 Current 能力，不再是"目标态"。涉及"用户可控删除"叙事的，仍以"保留期外可申请"为准。

**核心定位**：

- 不是替代人工规划师，而是**AI赋能的规划助手**
- 不是简单的信息查询，而是**智能决策支持**
- 不是通用AI对话，而是**垂直领域的专业服务**

### 2.2 目标用户

#### 主要用户画像

**用户A：普通工薪家庭考生家长**

- 年龄：40-50岁
- 职业：工薪阶层/小企业主
- 痛点：信息不对称，不知道如何选，怕选错
- 需求：专业建议、风险提示、操作简单
- 付费意愿：中等（几百元以内可接受）

**用户B：县城/农村考生**

- 信息获取渠道有限
- 对政策不了解
- 需要标准化的、可靠的建议
- 付费意愿：低（倾向于免费或低价）

**用户C：一线城市知识型家长**

- 有信息获取能力
- 但缺少系统化的方法论
- 需要数据支持和验证工具
- 付费意愿：高（为专业服务付费）

#### 用户场景

1. **出分第一时间**：刚查到分数，不知道能上什么学校
2. **填报前焦虑**：方案不确定，怕滑档/退档
3. **多方案对比**：有几个选项，不知道哪个更好
4. **政策理解**：对"院校专业组""调剂"等概念不清楚

### 2.3 核心价值主张

| 价值维度 | 传统方式           | 本项目                          | 提升          |
| -------- | ------------------ | ------------------------------- | ------------- |
| 信息获取 | 手工查资料（数天） | 对话即得（分钟）                | 效率提升100x  |
| 政策合规 | 凭经验（易错）     | 自动检查（准）                  | 错误率降低90% |
| 数据精度 | 主观估算           | 2025年位次精确匹配              | 精度提升50%   |
| 个性化   | 标准化话术         | 四维匹配（兴趣+能力+家庭+就业） | 匹配度提升40% |
| 风险提示 | 口头上提醒         | 可视化风险报告                  | 遗漏率降低80% |

### 3.3 功能详情

#### F001：省份自动识别

**用户故事**：

> 作为家长，我希望系统能自动知道我是哪个省的，这样我就不用手动选择省份了。

**验收标准**：

- [x] 可以从对话中自动提取省份名称（如"我是湖南考生"→识别出"湖南"）
- [x] 支持省份简称识别（如"湘"→"湖南"）
- [x] 正确加载对应省份的规则
- [x] 未识别到时给出提示

**技术实现**：

```python
detect_province("我是湖南考生...") # 返回 "湖南"
```

#### F002：方案生成

**用户故事**：

> 作为考生，我希望系统能根据我的分数和性格，推荐适合我的专业和学校。

**验收标准**：

- [x] 支持兴趣测评（霍兰德6型）
- [x] 支持学科强弱评估
- [x] 支持家庭条件匹配
- [x] 输出三套方案（冲/稳/保）
- [x] 每个方案包含6个专业+排序建议

**流程**：

1. 收集信息（5分钟）
2. 分析画像（2分钟）
3. 生成方案（5分钟）
4. 自检输出（1分钟）

#### F003：规范检查

**用户故事**：

> 作为家长，我担心方案有错误导致孩子滑档，希望系统能自动检查。

**验收标准**：

- [x] 检测致命错误（如"45个学校"）
- [x] 检测严重错误（如主观概率）
- [x] 检测一般警告（如体检要求缺失）
- [x] 输出三级分类报告
- [x] 提供具体修正建议

**检查项**：

- 志愿单位是否正确
- 调剂范围是否正确
- 数据是否基于2025年
- 是否标注数据来源

[对话继续]...

## 9. 业务场景与商业化设计

### 9.1 两种核心业务流程（关键区别）

> **核心原则**：闲鱼/微信流程 vs Web系统流程，有本质区别

| 维度         | 闲鱼/微信流程      | Web系统流程（目标态） |
| ------------ | ------------------ | --------------------- |
| **入口**     | 闲鱼/微信          | Web系统               |
| **资料提供** | 用户先提供完整资料 | 付费后填写            |
| **付费时机** | 渠道内先付费       | 系统内先付费          |
| **订单创建** | 管理端人工录入     | 系统自动创建          |
| **方案生成** | 管理端 → 龙老师    | 系统自动/人工         |
| **方案交付** | 闲鱼/微信发PDF     | 站内查看+邮件PDF      |

补充说明：当前已落地主链仍以闲鱼/微信/学校等人工服务运营链路为准；上表中的 Web 系统流程描述的是 T12 后续要收口的目标态闭环，不代表当前系统已经具备完整 Web 自助交付能力。

详细流程见 [BUSINESS_SCENE](../docs/BUSINESS_SCENE.md)

### 9.2 推广渠道矩阵

| 渠道              | 定位     | 业务流程                            | 适用版本             |
| ----------------- | -------- | ----------------------------------- | -------------------- |
| **闲鱼**          | 交易平台 | 先给资料→管理端录入→生成方案        | 基础版/标准版        |
| **小红书/知乎**   | 内容种草 | 引流→闲鱼/微信→先给资料→管理端录入  | 基础版/标准版        |
| **微信/企业微信** | 私域服务 | 先加微信→给资料→管理端录入→生成方案 | 基础版/标准版/尊享版 |
| **学校地推**      | 口碑传播 | 先加微信→给资料→管理端录入→生成方案 | 标准版/尊享版        |
| **Web系统**       | 自助服务 | 进系统→付费→填资料→生成方案         | 基础版/标准版/尊享版 |

### 9.3 订单管理

**两种订单来源**：

- **闲鱼/微信订单**：用户在渠道内先付款，运营人员在管理后台手动录入（含用户资料）
- **Web订单（目标态）**：系统自动创建订单（含用户填写的资料）

**统一管理**：

- 所有订单都进入管理后台
- 统一用户档案管理
