## 关键发现摘要

### 1. 项目真实定位
- **不是完整 SaaS**，是"管理后台 + 订单/分享/渠道同步 + AI 审核链路"的人工服务运营增强系统
- 用户端 Web 自助闭环仍**未收口**（虽然有 `admin/routes/web_public.py` 126KB 的 Python 内联 HTML MVP）
- 真实文档化的自我定位：`docs/REVIEW_REPORT_2026-06-19_SYSTEMATIC_PRODUCTION_READINESS_REVIEW.md` 给出 `documentation-forward, execution-moderate, production-lagging` 标签

### 2. 关键路径现状
- 覆盖率门槛 92.46% / core 100%（已达标）
- 三仓同步、CI 多 Python 版本、PDF runtime smoke、coverage gate 都已落地
- T12 用户端 Web 自助 MVP **本地主链大部分落地但线上未完成**
- 真实支付 acceptance、公网 notify_url、备案域名、生产告警/监控仍是硬缺口
- 2026-06-16 前端专项审计发现 3 个 P0：用户端开发术语外露、真实下单 500 失败、信任链缺失——首轮整改本地完成，剩余仅 Git 交付

### 3. 前端原型 vs 后端现状（关键决策依据）

| 维度 | `前端原型代码/`（Next.js 16 客户端原型） | `admin/routes/web_public.py`（FastAPI 现状） |
|---|---|---|
| 形态 | 客户端 SPA，Tailwind 4，React 19 | 服务端 Python 内联 HTML 渲染 |
| 后端对接 | **0** 个 fetch/axios 调用（src 下 grep 返回空） | 直接接 OrdersDAO/PaymentService |
| 订单/支付/资料 | 全部本地 mock | 真实端点：`POST /api/public/orders`、`/pay/mock/...`、`/portal/{token}/*` |
| 覆盖流程 | 6 页：对话/方案/咨询/方案对比/评估/关于 | 25+ 端点：`/`、`/pricing`、`/checkout/{v}`、`/portal/{token}/{info,status,report,...}` |
| 已有测试 | 无（前端原型独立目录，无 pytest/vitest 配置） | `admin/tests/test_web_public.py` 87/87 通过 |
| 部署形态 | Next.js dev/build，WorkBuddy 预览域白名单已配 | FastAPI + Docker Compose，已有 80+ 测试 |
| 规模 | 8 个页面 + 8 个 hooks + 4 个组件，1 个全局 layout | 1 个 126KB 文件 + 1 个 portal-ui.css + 1 个 dashboard.html |

### 4. 团队技术能力评估（基于代码证据）

**优势：**
- 后端：Python 治理、状态机、加密/脱敏、CI、覆盖率门禁、fail-closed 模式已成熟
- 测试：边界用例、契约测试、E2E、性能门禁、PDF 运行时都有
- 文档：真相源优先级、收口规则、API/架构/执行板分层做得比一般团队好

**短板：**
- 前端几乎空白：原型完全 mock，没有真后端对接，没有 e2e 测试，没有 lint/CI 接入
- `web_public.py` 126KB 是典型"模板堆代码"——继续维护会越来越难
- 没有设计系统、design token、统一组件库
- AI 痕迹重、模板化、品牌识别弱

### 5. 前端迁移结论（不建议原样照搬）

**直接迁移 Next.js 原型替代 FastAPI 页面 = ❌ 不可行**
原因：
1. 原型是 mock-only，不接后端，迁过去等于丢掉所有已完成的服务端能力
2. 后端 OpenAPI 体系已成形，迁移会重新打散事实源
3. 原型 Next 16 + React 19 还没在 CI/测试中验证过
4. 设计 token、design system、组件库都没有，迁移后还是"花架子"

**推荐路径：渐进式 React 化**
- 阶段 A：保留 FastAPI 真实端点 + 引入 OpenAPI TypeScript Codegen
- 阶段 B：在 `前端原型代码/` 中用 Next.js App Router 重写 6 个核心页面，对接后端真实 API
- 阶段 C：把 Next.js 嵌入到现有架构（独立部署或挂载到 FastAPI 反代之后）
- 阶段 D：建立 design token / 组件库 / Storybook / Playwright e2e
- 阶段 E：把 `admin/static/dashboard.html` + `web_public.py` 内部页面也 React 化

## 下一步建议（给 PM/lead）

1. 不要急于把 Next.js 原型推到生产；先把 6 个核心页面的真实 API 对接做出来
2. 先补 design token、组件库、Storybook，再谈视觉升级
3. 把 `前端原型代码/` 收编为 `apps/web-mvp/` 之类的子项目，纳入根目录的 monorepo
4. 给前端补 CI（typecheck + lint + build + e2e smoke）
5. 不要因为 Next.js 原型"看起来好看"就替换已有的 FastAPI 页面——治理成熟度差太多
