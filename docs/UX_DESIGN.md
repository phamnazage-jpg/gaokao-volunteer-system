## 2. 用户注册与登录策略

### 2.1 核心原则

> **"先服务，后注册"** - 降低使用门槛，先让用户感受到价值

### 2.2 咨询模式对比

| 模式         | 是否需要注册 |    功能限制     |   数据保存    | 适用场景           |
| ------------ | :----------: | :-------------: | :-----------: | ------------------ |
| **匿名咨询** |  ❌ 不需要   | 只能生成1个方案 |   ❌ 不保存   | 初次体验、快速了解 |
| **临时会话** |  ❌ 不需要   |    完整功能     | ⚠️ 仅本次会话 | 一次性咨询         |
| **注册用户** |   ✅ 需要    |    完整功能     |  ✅ 永久保存  | 长期跟踪、多次咨询 |

### 2.3 匿名咨询流程 (v2.0)

#### 设计说明

**当前版本(v2.0)完全支持不注册咨询：**

```
用户打开Hermes/命令行/Web
    ↓
直接输入："我是湖南考生，578分..."
    ↓
龙老师立即回应，开始咨询
    ↓
完整信息采集 → 方案生成 → 报告输出
    ↓
报告保存到本地 (/tmp/ 或当前目录)
    ↓
咨询完成，不强制注册
```

#### 匿名咨询限制

- ✅ 可完整体验所有功能
- ✅ 可生成可视化报告
- ⚠️ 报告仅保存在本地，不保存到云端
- ⚠️ 关闭会话后无法查看历史

## 3. 智能注册引导机制

### 3.1 自动注册触发条件

> **非强制，但智能引导**

#### 触发场景1：信息完整性达标

```
当用户提供了以下信息时：
- 姓名（或考生姓名）
- 手机号（或家长手机号）
- 省份
- 分数/位次

系统自动提示：
"您提供的资料已经很完整了，是否保存档案？
保存后您可以在任何设备上查看，下次咨询时不用重复填写。"
```

#### 触发场景2：报告生成完成后

```
报告生成后提示：
"✅ 您的志愿方案已生成！

💾 保存到云端，方便随时查看？
• 手机、电脑都能看
• 可分享给家人
• 填报前可再次咨询修改

[立即保存]  [仅本地保存]"
```

#### 触发场景3：主动提示补充信息时

```
当咨询过程中发现信息缺失：
"为了更好地为您规划，建议补充以下信息：
□ 考生姓名（用于生成报告）
□ 手机号（方便接收填报提醒）
□ 体检情况

补充后自动为您创建档案，下次登录可继续查看。"
```

### 3.2 注册流程设计

#### 极简注册 (3步完成)

```
Step 1: 确认基本信息（已预填）
━━━━━━━━━━━━━━━━━━━━
姓名：李明 (已填写)
手机号：138****8888 (已填写)
省份：湖南 (已填写)

[信息正确]  [修改]

Step 2: 设置密码
━━━━━━━━━━━━━━━━━━━━
设置登录密码（至少6位）：
[__________]

确认密码：
[__________]

💡 提示：密码用于查看档案，请牢记

Step 3: 完成
━━━━━━━━━━━━━━━━━━━━
✅ 注册成功！

您的档案号：GK20260611001

您可以通过以下方式查看：
• Web端：访问 gk.example.com
• 直接回复：查看我的方案
```

#### 密码策略

- 最小长度：6位
- 复杂度：不强制（用户自己选择）
- 找回方式：手机号验证码

### 3.3 软注册机制

#### 概念

> 先创建临时档案，用户确认后再升级为正式账户

#### 流程

```
1. 用户开始咨询
2. 系统自动生成临时档案ID (如：TEMP_20260611_001)
3. 咨询过程中自动保存信息到临时档案
4. 咨询完成时提示："是否保存档案？"
5. 用户确认后，临时档案升级为用户档案
6. 引导设置密码完成注册
```

#### 优势

- 不中断咨询流程
- 用户感受到"已有数据"，更愿意注册
- 注册过程顺滑

## 4. 主动信息补充提示机制

### 4.1 信息分级

| 级别     | 信息项                | 必要性 | 缺失提示                     |
| -------- | --------------------- | :----: | ---------------------------- |
| **必需** | 省份、分数/位次、选科 | ⭐⭐⭐ | 必须提供，否则无法咨询       |
| **重要** | 兴趣类型、学科强弱    | ⭐⭐⭐ | 缺少时影响方案质量，主动询问 |
| **有用** | 家庭情况、目标城市    |  ⭐⭐  | 完善后方案更精准，适时提示   |
| **可选** | 体检情况、特殊需求    |   ⭐   | 仅在相关时询问               |

#### 完善度评分

```
信息完善度：75% (重要信息已收集)

已提供：✅
• 省份、分数、选科
• 兴趣类型

建议补充：⚠️
• 学科强弱（影响专业推荐精准度）
• 家庭资源（影响院校层次建议）

点击下方快速补充：
[补充学科强弱] [补充家庭情况] [跳过，直接生成方案]
```

### 4.2 主动提示策略

#### 策略1：适时询问 (Timing-based)

```python
# 对话过程中，适时插入
if not has_strong_subjects:
    if conversation_turn >= 3 and random() < 0.3:
        prompt("对了，您孩子最擅长哪两门学科？这对我推荐专业很重要。")

if not has_family_info:
    if conversation_turn >= 5:
        prompt("顺便问一下，家里是普通工薪还是有一定资源？")
```

#### 策略2：方案生成前检查 (Pre-generation check)

```
龙老师：
"信息收得差不多了，我再确认几个关键点，确保方案更精准：

• 物理和数学是孩子的强项吗？
• 家里希望孩子毕业后直接工作还是考研？
• 有没有明确不想去的城市？

[快速回答] [直接生成方案]"
```

#### 策略3：报告中的待完善标注

```
报告页脚：
━━━━━━━━━━━━━━━━━━━━
📌 此方案基于以下信息生成：
✅ 省份、分数、位次
✅ 选科组合
✅ 兴趣类型
⚠️  学科强弱（未提供，使用默认值）
⚠️  家庭情况（未提供，使用典型值）

[补充信息，优化方案] [保持原方案]
```

### 4.3 非打扰式设计

#### 原则

> 不强制、不频繁、有价值时才问

#### 实现

- ✅ 最多每3轮对话询问1次
- ✅ 提供"跳过"选项
- ✅ 说明询问原因（为什么需要）
- ✅ 给出快速选择（减少输入成本）

```
Day 0: 初次接触
├── 渠道：朋友推荐/搜索/广告

└── 动作：开始第一次咨询

├── 多次咨询
```

## 8. 关键决策记录

### Decisions

#### D001: 是否支持完全匿名咨询？

**决策**: ✅ 支持
**理由**: 降低使用门槛，让用户先体验价值
**影响**: 需要在本地保存报告，不能云端同步

#### D002: 注册是否强制？

**决策**: ❌ 不强制，智能引导
**理由**: 尊重用户选择，体验优先
**实现**: 多次触发点引导，但不阻断流程

#### D003: 手机号是否必需？

**决策**: ⚠️ 推荐但不强制
**理由**: 重要但不阻断咨询
**用途**: 接收填报提醒、密码找回、个性化服务

#### D004: 主动提示频率？

**决策**: 最多3轮对话1次
**理由**: 平衡信息完整度和用户体验
**实现**: 带"跳过"选项，说明原因

#### D005: 是否做小程序？

**决策**: ❌ 不做
**理由**: 小程序限制多（审核、功能受限、扩展困难）
**替代方案**: Web端响应式设计 + PWA，覆盖所有用户场景

#### D006: Web端 vs 原生App？

**决策**: 优先 Web端，按需启动 App
**理由**: Web端开发快、跨平台、用户访问门槛低
**App启动条件**: Web端用户量大、有推送/深度集成需求时

## 9. 常见问题

### Q: 不注册能完整咨询吗？

**A**: ✅ 可以。当前版本(Hermes/命令行)已完全支持不注册咨询，所有功能可用，只是报告仅保存在本地。

### Q: 注册后有什么好处？

**A**:

- 方案永久保存，随时查看
- 跨设备同步（手机/电脑）
- 可分享给家人
- 接收填报提醒
- 多次咨询对比

### Q: 手机号会用于营销吗？

**A**: ❌ 不会。仅用于：密码找回、填报提醒（可选）、紧急联系。

### Q: 不想被频繁询问补充信息怎么办？

**A**: 每次询问都有"跳过"选项。也可一次性回复："使用默认值"，后续不再询问。

### Q: 未来会做App吗？

**A**: 计划2028年启动，前提是Web端用户量稳定、用户有深度集成需求。Web端是主战场。

### Q: 为什么不做小程序？

**A**:

- 小程序审核严格，政策相关功能可能受限
- 功能受限（不能做复杂逻辑）
- 跨平台体验不一致
- 维护成本高
- Web端已经能覆盖所有用户场景
