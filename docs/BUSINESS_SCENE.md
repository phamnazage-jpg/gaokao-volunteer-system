## 关键业务流程区分

> **核心原则**：闲鱼/微信流程 vs Web系统流程，有本质区别

| 维度         | 闲鱼/微信场景                 | Web系统场景                  |
| ------------ | ----------------------------- | ---------------------------- |
| **入口**     | 闲鱼下单 / 加微信             | 访问Web系统                  |
| **资料提供** | 用户先提供完整资料            | 用户先看到系统               |
| **付费时机** | 闲鱼/微信先付费               | 系统内付费                   |
| **生成方案** | 管理端人工录入资料 → 生成方案 | 系统自动/人工处理 → 生成方案 |
| **订单创建** | 管理后台录入订单（已支付）    | 系统自动创建订单             |
| **交付方式** | 闲鱼/微信发PDF                | 站内查看 + 邮件PDF           |
| **适用场景** | 人工服务、深度咨询            | 自助服务、标准流程           |

## 1. 推广场景总览

### 1.1 渠道矩阵

```
                     线上                    线下
                    ┌───┐                  ┌───┐
         付费流量   │ 闲鱼│                  │   │
                    │  │  │                  │ 学校│
         内容种草   ├───┼───┤    口碑传播    │   │
                    │小红书│◄────────────────│师生│
         知识营销   │  │  │                  │   │
                    │ 知乎│                  │   │
                    └───┘                  └───┘
```

### 1.2 渠道定位

| 渠道       | 定位     | 交易地点         | 服务方式                 | 资料流程                        |
| ---------- | -------- | ---------------- | ------------------------ | ------------------------------- |
| **闲鱼**   | 交易平台 | 闲鱼内交易       | 闲鱼发方案 或 引流至微信 | 先给资料→付费→管理端录入→发方案 |
| **小红书** | 内容种草 | 引流至闲鱼或微信 | 微信交付                 | 先给资料→付费→管理端录入→发方案 |
| **知乎**   | 知识营销 | 引流至闲鱼或微信 | 微信交付                 | 先给资料→付费→管理端录入→发方案 |
| **学校**   | 口碑传播 | 微信/企业微信    | 微信交付                 | 先给资料→付费→管理端录入→发方案 |
| **Web端**  | 自助服务 | Web内支付        | 系统自动生成报告         | 进系统→付费→填资料→生成方案     |

#### 关键说明

**为什么这样设计？**

- 闲鱼/微信是「人工服务场景」，用户期望「先沟通再给资料」
- 管理端录入是「桥梁」，将线下成交转化为系统数据
- 保证所有渠道数据统一进入系统，便于统计和管理

**管理端录入的字段**：

```
订单信息：
├── 来源渠道：闲鱼/微信/学校/转介绍

#### Web系统设计要点

**为什么先付费？**

- Web系统是「标准产品」，用户浏览后即可决策
- 自助服务需要付费门槛，筛选真实需求
- 付费后才解锁资料填写，避免无效数据

**付费前的展示内容**：

- 服务案例（脱敏版）
- 服务流程说明
- 价格清晰透明
- 用户评价（真实评价）

**付费后的体验**：

- 清晰的资料填写引导
- 进度提示（步骤1/3）
- 预计完成时间提示
- 完成后多渠道通知（站内+邮件+微信）

```
创建订单 ──→ 待服务 ──→ 资料补充 ──→ 方案生成 ──→ 已交付 ──→ 已完成

**状态说明**：
| 状态 | 说明 | 操作 |
|------|------|------|
| **草稿** | 订单信息未完整 | 补充信息 |
| **待服务** | 已分配顾问 | 等待处理 |
| **资料补充** | 缺少关键信息 | 联系用户补充 |

| **方案生成** | 生成PDF中 | - |
| **已交付** | PDF已发送 | 等待确认 |
| **已完成** | 服务结束 | - |
| **待退款** | 用户申请退款 | 审批退款 |

## 4. 两种模式的对比总结

|   对比维度   |      闲鱼/微信模式       |   Web系统模式   |
| :----------: | :----------------------: | :-------------: |
| **用户决策** |     先沟通了解再下单     | 浏览后直接下单  |
| **资料收集** | 用户主动提供（聊天记录） |  系统表单收集   |
| **付费时机** |     闲鱼/微信先收款      |    Web内支付    |
| **订单创建** |     管理后台手动录入     |  系统自动创建   |
| **资料录入** |     运营人员复制粘贴     |  用户自主填写   |
| **方案生成** |      龙老师人工生成      | AI自动/人工辅助 |
| **交付方式** |     闲鱼/微信发文件      |    站内+邮件    |
| **适用用户** |    需要深入沟通的家长    | 明确需求的家长  |
| **服务深度** |      高（深度咨询）      | 中（标准流程）  |
| **运营策略** |     主推（信任度高）     | 辅助（规模化）  |

## 5. 运营SOP

### 5.1 闲鱼订单处理SOP

```
用户拍下商品
    ↓
10分钟内响应
    ↓
发送信息收集模板：
"亲，请提供以下信息，我为您安排龙老师服务：
1. 省份：
2. 分数：
3. 选科：
4. 联系电话：
5. 孩子平时擅长什么学科？"
    ↓
用户提供信息
    ↓
复制信息 → 管理后台【新建订单】
    ↓
选择来源：闲鱼
填写：订单号、金额、客户信息、考生信息
    ↓
点击【保存并生成档案】
    ↓
分配给龙老师（系统自动或手动）
    ↓
告知用户：
"已为您安排龙老师，预计1小时内交付方案，请留意闲鱼消息~"
    ↓
龙老师完成方案
    ↓
通过闲鱼发送PDF
    ↓
跟进：
"方案已发送，请查收。有任何疑问随时联系，7天内可免费修改一次~"
    ↓
用户确认收货
    ↓
管理后台标记【已完成】
```

### 5.2 微信订单处理SOP

```
用户加微信
    ↓
自动欢迎语 + 信息收集表
    ↓
用户提供信息（或语音沟通）
    ↓
报价确认 → 用户转账/扫码支付
    ↓

    ↓
复制信息 → 管理后台【新建订单】

    ↓

    ↓

    ↓
管理后台标记【已完成】
```

### 5.3 数据一致性检查

**每日核对项**：

```
□ 闲鱼成交单数 = 管理后台闲鱼来源订单数
□ 微信收款笔数 = 管理后台微信来源订单数
□ 未完成订单跟进（超过2小时未交付）
□ 待分配订单处理（超过30分钟未分配）
```

## 6. 异常处理

### 6.1 信息不完整

**场景**：用户只给了分数，没给选科

**处理**：

```
管理端标记：【资料补充】
↓
联系用户：
"您还需要提供选科组合（如物理+化学+生物），
我们才能生成完整方案~"
↓
用户补充
↓
更新订单信息
↓
继续服务流程
```

### 6.2 退款处理

**闲鱼退款**：

- 按闲鱼平台规则处理
